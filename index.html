04 / Technical Deep Dive

Configuration as the contract.

A flagship pattern across both platforms: analytical frameworks defined in YAML, not code. Every agent reads the same config — datasets, hierarchies, column mappings — so domain rollouts ship without code changes.

frameworks/sell_out_analytics.yaml YAML 
framework: sell_out_analytics
version: "3.2.0"
owner: tech-lead@decisionpoint

datasets:
  - id: pos_daily
    grain: [date, store, sku]
    measures: [units, value, on_promo]
  - id: master_product
    grain: [sku]
    columns: [brand, category, segment]

hierarchies:
  product: [category, segment, brand, sku]
  geo:     [region, state, city, store]
  time:    [year, quarter, month, week, date]

prompts:
  router:    prompts/router.v4.j2
  planner:   prompts/planner.v6.j2
  validator: prompts/validator.v2.j2

guards:
  max_tokens_per_question: 12000
  require_citation: true
  block_on_grounding_fail: true

Why config-as-contract?

  1. Domain velocity. A new client domain ships in days, not sprints — change YAML, not code.
  2. Auditable behavior. Every prompt version, token cap, and validation rule is in version control.
  3. Cost discipline. Token budgets are declared, not implicit. PRs surface cost changes upfront.
  4. Safe rollouts. Guards (citation, grounding) are required by the framework, not optional code paths.
